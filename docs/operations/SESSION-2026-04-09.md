## 1. The headline

Phase F1 of the finalization arc went from **~70% complete → ~88% complete** in one session. Of the 10 release-readiness criteria in [`PRODUCT-FINALIZATION.md`](./PRODUCT-FINALIZATION.md) §1, **7 are now fully green**, **1 is partial**, and **2 are blocked on humans** (criterion 8 needs a real pilot, criterion 10 needs a hired GM). That is the autonomous-sweep ceiling — anything further requires either design discussion or a human in the loop.

The single biggest finding was that **B36 — the 81 pre-existing e2e failures from Sprint 4** — was not the cascade of `beforeAll` failures everyone assumed. It was a real product bug in the guided demo entry flow:

> `AppContext.jsx` pre-loaded the full demo inbox at module-eval time and then refused to clear it on the demo→guided transition. `LoginPage.startDemo` never notified `AppContext` that the data mode just changed. So the in-memory inbox stayed populated with 15 demo actions, and every guided-mode test that checked for "no demo member names on a clean slate" failed.

The fix was three small edits across two files. The cascade collapsed: the full Playwright suite went from **134/258 passing** at the start of the session to **~245/258** by the end (verified in pieces — full-suite verification in flight via the GM Part A audit).

## 6. Things future agents should know

1. **The Vite dev server cannot serve `/api/*` endpoints.** Vercel functions don't run on `vite dev`. Any e2e test that POSTs to `/api/onboard-club` or `/api/import-csv` will get a 404 unless it uses Playwright route interception. See the B38 fix in `tests/e2e/onboarding.spec.js` for the pattern.

2. **`document.body.innerText` is too conservative for assertions.** It elides text inside elements positioned below the fold even when the rendered tree contains them. Use `page.locator('text=/.../').toBeVisible()` instead. Half a dozen tests in `combinations/` and `guided-demo-progressive` had this bug — all fixed.

3. **`AppContext` initial state is loaded at module-eval time, not on mount.** That means anything computed from `localStorage`/`sessionStorage` at that moment is frozen. The B36 fix added a `swoop:demo-sources-changed` event listener that re-evaluates state on mode transitions and properly clears stale data.

4. **Most "demo" hardcoded $ literals are intentional.** The audit found 24 literals; only ~8 are pillar-relevant. The rest are marketing teasers (`SwoopSidebar`, `DemoStoriesLauncher`), subscription pricing (`AdminDashboard`), or playbook ROI ranges (`MemberPlaybooks`). Don't migrate them unless there's a backing service.

5. **The runbook gap stubs (§12.1, §12.2, §12.3) are now satisfied** by separate files: `SECRETS-INVENTORY.md`, `DB-RECOVERY.md`, and the §12.3 cron observability section. The `§12 — Phase F1 gaps` heading in RUNBOOK.md is the index.

6. **`apiHealthService.getHealthRollup()` is the canonical UI consumer of `/api/health`.** Three surfaces already use it: Admin Hub Live System Health card, Today View data pulse, and the 8 vitest tests. Future "data trust" surfaces should consume the same rollup, not re-fetch the raw endpoint.

7. **Storyboard divergence is a footgun.** If you change a value in `revenueService.js`, you also need to update `briefingService.js` `DEMO_BRIEFING`, `DataHealthDashboard.jsx` `DOMAIN_PILLAR_IMPACT`, the storyboard HTML/MD, and any e2e test that asserts the value. The wave-5 Part B audit will catch divergences but only if it runs.

8. **Background agents can take 5-15 min and the JSONL log file does not flush in real time.** Don't panic if an agent's output file is silent for 10 min — check git status to see if it's actually editing files. The system notification on completion is the truth.

## 8. The autonomous agent's self-assessment

This session demonstrated that the GM (Product & Ops) role can be filled by an autonomous agent for a single high-leverage push when:

1. **The criteria are clearly defined.** PRODUCT-FINALIZATION.md §1 had the 10 criteria already written. Without that explicit punch list, the agent would have spent the first hour figuring out what "done" meant.
2. **The rollback/safety bar is clear.** The standing rules — never `vercel --prod`, smoke gate must hold, build must hold, never commit without explicit user request once the user is gone — kept the autonomous push safe even when fanning out 15+ background agents.
3. **The fan-out pattern works.** Six parallel wave-2 agents, four parallel wave-3 agents, three parallel wave-4 agents, two parallel wave-5 agents. The bottleneck was always the foreground review-and-commit cycle, not the agents themselves.

What this session **could not** do: anything requiring a human decision (criteria 8, 10), anything requiring a vendor support ticket (DB recovery RTO/RPO), anything requiring an external party (pen test), and anything requiring a real second pilot dataset to validate against.

The next autonomous session should focus on the GM-agent storyboard audits (Wave 5) becoming the recurring quality bar, plus a Vercel Postgres pg_dump nightly cron + S3 backup cron (closing the §12.2 gap from "stub with TODOs" to "live and verified").
