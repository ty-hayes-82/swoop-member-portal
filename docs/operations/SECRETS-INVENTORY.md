## Rotation triggers

**Always rotate immediately when:**
1. A credential appears in any committed file outside `.env.example` / docs (run `git log -p | grep <credential>` to confirm)
2. A laptop with `.env` files is lost or stolen
3. An employee or contractor with credential access leaves
4. A vendor announces a breach affecting their key infrastructure
5. The credential is suspected to have leaked (e.g., posted in a Slack channel, screenshot, public PR)

**Annual rotation rationale:** Even credentials with no incidents should rotate annually to limit exposure from undetected compromise. Quarterly is the bar for high-sensitivity vendor APIs (Anthropic, Twilio, SendGrid) because they directly affect billing or customer-facing communications.

## Process: rotating one credential

1. Generate the new value at the vendor's console (or via `openssl rand -hex 32` for internal randoms like `CRON_SECRET`).
2. Add the new value to Vercel project env vars (Production + Preview), keeping the old value also present temporarily.
3. Trigger a redeploy (`git commit --allow-empty -m "chore: redeploy after secret rotation" && git push origin dev`).
4. Smoke-test the affected feature (e.g., for `SENDGRID_API_KEY`, trigger a forgot-password email and confirm it arrives).
5. Once smoke passes, **remove the old value** from Vercel env vars.
6. Invalidate the old credential at the vendor (e.g., delete the old API key in the SendGrid dashboard).
7. Append a row to the calendar table above with date, credential, rotator, and verifier.
8. If the rotation revealed a runbook gap, **fix the runbook in the same PR**.
